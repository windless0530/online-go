# online-go

一个在线围棋对弈项目。单人全栈开发，前后端分离。

## 项目结构

- 前端与后端代码物理隔离，分别位于独立目录（建议 `frontend/` 与 `backend/`），各自独立构建、独立运行。
- 前后端通过明确定义的 HTTP/WebSocket 接口通信，接口契约（请求/响应类型）需在两端保持同步。
- 当前阶段不引入数据库，状态保存在后端进程内存中；所有涉及持久化的设计需预留接口边界，未来替换为数据库时不影响上层逻辑。

## 前端

- **语言**：TypeScript 为唯一开发语言，禁止使用纯 JavaScript（构建产物除外）。
- **类型**：开启 `strict` 模式，禁止 `any`；外部数据进入边界时必须做类型校验或显式断言并附说明。
- **CSS**：样式独立成文件，禁止内联 `style=` 与运行时拼接样式字符串。组件与样式同名同目录共存（如 `Board.tsx` + `Board.css`）。
- **命名**：组件 `PascalCase`，函数与变量 `camelCase`，常量 `UPPER_SNAKE_CASE`，文件名与默认导出符号保持一致。
- **副作用**：UI 与业务逻辑分层，网络请求与状态管理不写入展示组件内部。

## 后端

- **语言**：Go。遵循官方 `gofmt`、`go vet`，并启用 `golangci-lint`（至少包含 `errcheck`、`govet`、`staticcheck`、`ineffassign`、`gosimple`）。
- **错误处理**：所有 `error` 必须显式处理，禁止 `_ = err` 丢弃；错误向上传递时使用 `fmt.Errorf("...: %w", err)` 包装上下文。
- **包结构**：按领域而非按层组织（如 `game/`、`match/`、`server/`），避免 `utils/`、`common/` 这类无语义包名。
- **接口边界**：对外暴露的 handler 与内部领域逻辑分离；领域逻辑不依赖 `net/http`。
- **并发**：明确每条 goroutine 的生命周期与退出路径，使用 `context.Context` 控制取消，禁止裸 `go func()` 无超时无取消。
- **内存状态**：当前以内存保存对局/房间状态，访问需有锁或单 goroutine 串行化，约定后续替换为持久化层的接口。

## 代码质量与稳健性

- **小步提交**：每个 commit 只做一件事，commit message 描述「为什么」而非「做了什么」。
- **不写无用代码**：不为假想的未来需求添加抽象、配置项、feature flag；三处相似优于过早抽象。
- **不写无用注释**：命名能说清的事不写注释；只有「为什么这样做」「不显然的约束」才注释。
- **不吞错误、不写无意义兜底**：内部代码信任类型与不变量，只在系统边界（用户输入、外部接口）做校验。
- **测试**：核心规则（围棋落子、提子、判劫、数子）必须有单元测试；前端关键交互逻辑可测部分需脱离 DOM 编写纯函数测试。
- **改动验证**：UI 变更必须在浏览器中实际操作验证；后端逻辑变更需运行相关测试通过后再提交。

## 协作约定

- 提交前需通过：前端 `tsc --noEmit` + lint + 测试；后端 `go build ./...` + `go vet ./...` + `golangci-lint run` + `go test ./...`。
- 涉及前后端接口变更时，同一个 commit 内同步两端，避免中间状态不可运行。
- 重构与功能变更分离提交，不在功能 commit 中夹带无关清理。
